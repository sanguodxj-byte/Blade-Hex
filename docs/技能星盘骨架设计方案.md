# 技能星盘骨架设计方案（草案）

> 目标：把技能星盘从"程序随意生成布局"改为"规范的、数据驱动的固定骨架 + 内容填空"。
> 本文是**设计方案**，不含实现代码。供评审，定稿后再实施。

---

## 一、已确认的设计决策（来自评审）

| 维度 | 决策 | 理由 |
|------|------|------|
| 布局来源 | **外部配置文件**驱动，彻底替换 `AutoLayoutAndConnect` | 现状程序乱生成、不可控 |
| 节点位置表达 | **显式存每节点占用的所有瓦片坐标** | 镶嵌宝石需精确密铺，BFS 自动展开做不到 |
| 连通机制 | **统一为几何瓦片共边**，废弃 `Neighbors` 列表 | 单一真相源，布局即连通 |
| 美学原则 | **镶嵌宝石（单元自美）** | 玩家跨扇区零散点亮，整体造型路线（像素画）会"点不满即残缺"，与"鼓励多属性"冲突 |
| 连通结构 | **六主干放射 + 相邻区过渡桥连** | 既有主干骨架，多属性又能横跨 |
| 半径/密度 | 后续填充阶段再定 | 先定几何骨架结构 |
| 大节点成本 | 统一 **大三角 = 4 瓦片**（被动大节点现为 3，是待修偏差） | 评审口径 |

### 关键设计洞察（否决像素画的原因）
游戏**鼓励多属性/复合职业** → 玩家跨多个扇区各点一部分 → 若每扇区是一幅完整造型（剑/弓/法杖），多属性玩家看到的是六个残缺半成品。像素画只奖励"专精一路"，与机制矛盾。
**镶嵌宝石**则相反：美感来自每个节点自身是一块完整宝石，点亮任意子集都是"一片镶嵌的宝石"，无论跨几个扇区都自洽。

## 二、几何模型

### 2.1 三角瓦片坐标系（沿用现有 `SkillTreeCoord`）
- 每个 hex 格点 `(q,r)` 关联 2 个三角瓦片：`t=0` ▽、`t=1` ▲
- 瓦片编码 `EncodeTile(q,r,t) = Vector2I(q*2+t, r)`
- 共边邻接：每个三角瓦片恰好 3 个共边邻居（`GetTileNeighbors`）
- C# 端与 `tools/skill_editor.html` 坐标系**已完全一致**

### 2.2 宝石单元（节点形状）
每个节点 = 一组**连续共边**的三角瓦片，构成一块"宝石"：

| 节点类型 | 瓦片数 | 形状 | 几何 |
|---------|-------|------|------|
| 小节点 | 2 | 菱形 | 1 个 ▽ + 1 个相邻 ▲ 拼成菱形 |
| 大节点 | 4 | 大三角 | 4 个小三角拼成一个 2× 边长的大三角（标准三角剖分：3 角 + 1 中心倒三角） |
| Keystone | 6 | 六角星/王冠 | 6 个三角绕一个 hex 格点一圈（= `GetStartTiles` 的形态） |
| start | 6 | 中心六边形 | 同 Keystone 形态，居原点 |

**密铺约束**：相邻宝石必须**恰好共享三角边、互不重叠**。这要求配置里每个节点显式列出其占用瓦片，且全局保证：
1. 无两节点占用同一瓦片（不重叠）
2. 想连通的两节点，至少有一对瓦片共边（连通）

## 三、骨架结构（六主干放射 + 过渡桥连）

```
                    INT区
          WIS区  ╲   │   ╱  CON区
                  ╲  │  ╱
                   ╲ │ ╱
        ───────────start───────────
                   ╱ │ ╲
                  ╱  │  ╲
          CHA区  ╱   │   ╲  STR区
                    DEX区
```

- **start** 居中（原点六边形），6 条边各对接一条主干。
- **6 主干**：每条主干从 start 出发，沿一个 60° 扇区向外延伸，是一条**连续共边的瓦片链**（宝石串）。主干上交替/分布小节点与大节点，keystone 在主干末端（最外圈）。
- **过渡桥**：相邻两扇区之间，在边界放置过渡节点（现有 `trans_*`），其瓦片同时与两侧扇区的某节点共边 → 实现横跨。
- **分支**：主干可在中途分叉出支链（如现有 STR 的剑格/剑刃分宽），但分支也是宝石串，遵循同样的共边密铺。

### 每区槽位预算（当前节点数，供参考；数据将大改）
| 区 | 小节点 | 大节点 | keystone |
|----|-------|-------|---------|
| STR | 21 | 8 | 1 |
| DEX | 19 | 13 | 1 |
| CON | 15 | 8 | 1 |
| INT | 17 | 10 | 1 |
| WIS | 14 | 9 | 1 |
| CHA | 15 | 9 | 1 |
| 过渡 | 19 | 0 | 0 |

> 注：各区节点数不等（DEX 大节点最多），骨架按区裁剪，不强求对称。

## 四、配置文件格式（草案）

载体：JSON，放 `res://BladeHexCore/src/SkillTree/skill_tree_layout.json`（或拆分 `_layout` 布局 + `_content` 内容两份）。
加载模板照搬 `SkillRegistry.cs`（Godot `Json`+`FileAccess`，Core 可用）。

### 4.1 布局 + 内容合一格式
```json
{
  "version": 1,
  "hexRadius": 12,
  "nodes": [
    {
      "id": "str_s01",
      "region": "str",
      "type": "small",            // small | big | keystone | start
      "tiles": [[2,0,0],[3,0,1]], // 显式瓦片列表 (q,r,t)
      "name": "强健体魄",
      "effect": "",               // 大节点的 skillEffect；小节点空
      "isActiveSkill": false,
      "statBonuses": { "max_hp": 3 },
      "description": "",
      "requiredLevel": 0,
      "keystoneCost": "",         // 仅 keystone
      "costBonuses": {}           // 仅 keystone
    }
  ]
}
```

**连通性不在 JSON 里显式写** —— 由加载后对 `tiles` 做几何共边计算自动得出（呼应"统一几何共边"）。

### 4.2 布局/内容分离（可选，推荐）
- `skill_tree_layout.json`：只含 `id / region / type / tiles`（几何骨架，改动频率低）
- `skill_tree_content.json`：只含 `id / name / effect / statBonuses / ...`（内容填空，改动频率高）
- 好处：调数值不碰布局，调布局不碰数值。

## 五、实现改造点（定稿后执行）

1. **SkillNodeData**：新增 `ExplicitTiles: Vector2I[]` 字段；`GetTiles` 优先返回它。
2. **SkillNodeShape.GetTiles**：`if (node.ExplicitTiles?.Length > 0) return node.ExplicitTiles;` 否则回退现有模板展开。
3. **新建 SkillTreeLayoutLoader**（Core）：照 `SkillRegistry` 模式从 JSON 读节点 → 构造 `SkillNodeData`。
4. **SkillTreeData 构造**：改为 `LoadFromJson()`，删除 `AutoLayoutAndConnect` + 六个 `BuildXxxRegion` + `ConstellationBuilder` 调用（彻底替换）。
5. **连通**：`NodeFiller` 已基于几何共边，无需改；`Neighbors` 字段废弃（保留字段定义但不再填充，或彻底删除引用）。
6. **大节点瓦片数**：`GetRequiredTileCount` 被动大节点 3→4（与"大三角=4"对齐）。
7. **前端**：零改动（已确认前端完全依赖 GridPosition/GetTiles，无硬编码）。
8. **瓦片成本/点数**：点亮一个节点消耗 = 其瓦片数（小2/大4/key6），沿用现有 `FillNodeTile` 逻辑。

### 风险与待解
- **谁来生成那份规范布局 JSON**：这是最大工作量。骨架几何要人工设计每节点的瓦片坐标，保证全局不重叠+连通。需要一个**生成/校验工具**（可后续做一个 C# 校验器：加载 JSON → 检测瓦片重叠、连通性、是否在六边形内）。
- **数据大改**：节点数量/效果将变，骨架要先定"槽位结构"再填内容，避免反复重排。
- **start 与主干的对接**：start 6 瓦片的 6 条外边各对接一条主干首节点，需保证共边。
