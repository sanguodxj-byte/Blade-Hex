# 存档系统

## 概述

本游戏的存档系统需要处理**永久死亡**带来的独特挑战——角色一旦死亡不可复活（除极少数高阶法术），因此存档策略直接影响游戏体验。系统需在"容错性"与"后果感"之间取得平衡。

当前代码基础：`SaveManager.gd` 已实现单存档位的基础序列化（经济/角色/迷雾数据），本设计将其扩展为完整的多槽位存档系统。

---

## 一、存档模式

### 1.1 标准模式

| 设置 | 说明 |
|------|------|
| 手动存档 | 10个槽位，玩家自由选择 |
| 自动存档 | 3个旋转槽位（覆盖最旧的） |
| 存档限制 | 战斗中不可存档，战斗地图为离散场景，不存在中途状态 |

### 1.2 铁人模式（Ironman）

| 设置 | 说明 |
|------|------|
| 存档槽位 | 仅1个自动存档槽，无手动存档 |
| 自动存档 | 与标准模式相同触发条件 |
| 死亡 | 永久。角色死亡不可读档挽回 |
| 开启时机 | 新游戏时选择，一旦开始不可更改 |
| 标记 | 存档UI显示骷髅图标，角色卡显示"铁人"标记 |

### 1.3 模式对比

| 维度 | 标准模式 | 铁人模式 |
|------|----------|----------|
| 手动存档 | ✅ 10槽 | ❌ |
| 自动存档 | ✅ 3槽旋转 | ✅ 1槽覆盖 |
| 战斗中存档 | ❌ | ❌ |
| 角色死亡 | 永久（可读档回退） | 永久（不可回退） |
| 退出不保存 | 可（读旧档） | 不可能（自动保存已覆盖） |
| 成就 | 标准 | 独有铁人成就 |

## 二、自动存档触发条件

| 时机 | 类型 | 说明 |
|------|------|------|
| 战斗结束 | 即时 | 含战利品分配确认后立即保存 |
| 进入城镇 | 即时 | 进入城镇界面时 |
| 离开城镇 | 即时 | 返回大地图时 |
| 接受委托 | 即时 | QuestManager状态变更后 |
| 完成委托 | 即时 | 领取奖励确认后 |
| 阶段晋升 | 即时 | 雇佣兵→骑士/骑士→领主 |
| 大地图每N天 | 定时 | 默认每10游戏日（可在设置中调整） |
| 英雄升级 | 即时 | 属性/技能点分配确认后 |
| 英雄死亡 | 即时 | 死亡结算完成后（无法回退） |
| 招募/解雇 | 即时 | 队伍成员变更时 |
| 领地建设 | 即时 | 建筑/升级确认后 |

> 注：`GameSettings.auto_save` 和 `GameSettings.auto_save_interval` 已在代码中实现。

## 三、存档数据结构

### 3.1 完整数据清单

存档文件使用 JSON+二进制混合格式：元数据用 JSON（便于展示），游戏数据用 Godot `store_var`（保留类型信息）。

```
存档文件结构:
├── metadata.json          # 存档元数据（快速读取，不加载完整存档）
└── save_data.bin          # 完整游戏数据（Godot 序列化）
```

### 3.2 元数据（metadata.json）

| 字段 | 类型 | 说明 |
|------|------|------|
| version | String | 存档格式版本（如 "0.3.0"） |
| timestamp | Dictionary | 保存时间（系统时间） |
| game_date | String | 游戏日期（如"1250年3月15日·春季"） |
| play_time_seconds | int | 累计游玩时间（秒） |
| phase | int | 当前阶段（1=雇佣兵, 2=骑士, 3=领主） |
| hero_count | int | 当前队伍英雄数量 |
| hero_names | Array | 英雄名字列表（用于存档槽显示） |
| gold | int | 当前金币 |
| level | int | 主角等级 |
| is_ironman | bool | 是否铁人模式 |
| screenshot_path | String | 缩略图路径（大地图截图） |

### 3.3 游戏数据（save_data.bin）

| 数据域 | 内容 | 对应代码 |
|--------|------|----------|
| **全局状态** | 游戏阶段、铁人标记、游玩时间 | `GlobalState` |
| **经济** | 金币、食物、天数、月、年、季节 | `EconomyManager` |
| **主角** | 全部 UnitData 字段（属性/特质/装备/技能盘/法术/坐骑/XP/等级） | `UnitData` |
| **队友列表** | 所有队友的 UnitData | `UnitData[]` |
| **军队** | 兵种堆叠列表（类型/人数/装备方案） | Phase 4 实现 |
| **背包** | 所有物品（武器/防具/消耗品/法术书/材料），含词缀 | `ItemData[]` |
| **大地图位置** | 玩家在大地图上的坐标 | `OverworldParty.position` |
| **战争迷雾** | 已探索格子网格（RLE压缩） | `FogOfWar.serialize()` |
| **AI实体** | 所有大地图AI实体（位置/状态/军队/目标） | `OverworldEntityManager` |
| **定居点** | 所有城镇/村庄/城堡状态（繁荣度/守军/所有权/设施） | `OverworldPOI[]` |
| **外族聚落** | 所有哥布林/狗头人/牛头人聚落状态 | `OverworldPOI[]` |
| **委托** | 已激活/已完成/可接取的委托 | `QuestManager` |
| **势力关系** | 各势力关系值 | Phase 5 实现 |
| **声望** | 总声望值 + 各势力声望 | 新系统 |
| **对话标记** | 已完成的对话ID/选择分支标记 | 新系统 |
| **世界事件** | 暗影教团进度、龙族觉醒状态、事件链 | 新系统 |
| **成就进度** | 已解锁/进行中的成就 | 新系统 |
| **设置快照** | 游戏设置（不与全局设置耦合） | `GameSettings` |

### 3.4 序列化策略

| 数据类型 | 序列化方式 | 说明 |
|----------|------------|------|
| 基础类型（int/float/String） | 直接存储 | Godot `store_var` 原生支持 |
| Resource（UnitData/ItemData等） | 转为 Dictionary | 避免路径依赖 |
| 大地图坐标 | Vector2i | 直接序列化 |
| 迷雾网格 | RLE 压缩数组 | 已在 `FogOfWar.serialize()` 中实现 |
| 物品词缀 | 词缀ID列表 | 反序列化时从词缀库查找重建 |
| 技能盘 | 节点ID列表 | 通过 `character_id` 关联全局 `SkillTreeManager` |

## 四、读档与恢复

### 4.1 恢复流程

```
读取 metadata.json → 校验版本兼容性
    │
    ▼
读取 save_data.bin → 反序列化为 Dictionary
    │
    ▼
重建场景状态:
    ├── EconomyManager: 恢复金币/食物/时间
    ├── UnitData: 恢复主角+队友属性/装备/技能盘
    ├── 大地图: 重建 HexOverworldGrid + 玩家位置
    ├── 战争迷雾: 恢复 explored_grid
    ├── AI实体: 重建所有 OverworldEntity
    ├── 定居点: 恢复所有 POI 状态
    ├── 委托: 恢复 QuestManager 状态
    └── 势力/声望: 恢复关系矩阵
    │
    ▼
进入大地图场景（OverworldScene）
```

### 4.2 场景恢复规则

| 存档时所在场景 | 读档恢复到 | 说明 |
|----------------|------------|------|
| 大地图 | 大地图（OverworldScene） | 直接恢复 |
| 城镇内部 | 大地图（该城镇位置） | 不保存在城镇内 |
| 战斗中 | ❌ 不存在 | 战斗中不可存档 |
| 主菜单 | ❌ 不存在 | 不在主菜单存档 |

### 4.3 战斗中的存档策略

**战斗中不可存档。** 战斗是离散事件——一旦进入战斗，必须完成（胜利/失败/撤退）后才能存档。这确保：
- 战斗结果不可回退（符合永久死亡哲学）
- 避免战斗中途状态的复杂序列化
- 战斗失败 = 读最近的自动存档（标准模式）

### 6.2 版本迁移

| 字段 | 说明 |
|------|------|
| version | 存档格式版本号（语义化版本，如 "0.3.0"） |

### 6.3 完整性校验

| 机制 | 说明 |
|------|------|
| CRC32 校验 | 写入时计算校验和，读取时验证 |
| 魔数校验 | 文件头包含固定魔数（0x5N1AD6E），快速判断文件类型 |
| 读取验证 | 反序列化后检查关键字段是否存在，缺失则标记为损坏 |
| 损坏处理 | 提示"存档已损坏"，提供删除选项（铁人模式除外） |

## 七、与永久死亡的关系

### 7.1 核心原则

永久死亡是本游戏的基石机制。存档系统必须**尊重死亡的永久性**：

- 英雄死亡时，立即自动保存（锁定死亡状态）
- 死亡英雄的数据标记为 `is_dead: true`，不可恢复
- 铁人模式下，死亡即后果——没有读档回退的可能
- 标准模式下，玩家可以读档到死亡前的自动存档，但这意味着**放弃死亡后的所有进度**

### 7.2 死亡后的选择

| 模式 | 死亡后 |
|------|--------|
| 标准 | 读档回到最近自动存档（死亡前），失去中间进度 |
| 标准 | 或接受死亡，继续游戏（招募新队友填补空缺） |
| 铁人 | 必须接受。自动保存已在死亡时覆盖 |

### 7.3 全灭处理

| 模式 | 全灭后 |
|------|--------|
| 标准 | 弹出"战败"界面 → 读档/返回主菜单 |
| 铁人 | 弹出"战败"界面 → 仅可返回主菜单/开始新游戏 |
