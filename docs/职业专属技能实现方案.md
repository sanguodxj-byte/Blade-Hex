# 职业专属技能实现方案

## 目标

把 `docs/职业专属技能.md` 中已经确认的 v1 分阶职业技能设计落实到游戏内，并补充单元测试，确保职业称号、职业技能元数据、主动技能、关键被动钩子和 UI 使用规则一致。

本方案只处理职业专属技能系统，不重做技能盘、不重做先攻队列、不重做命中体系。

## 已确认规则

1. 角色只获得当前职业称号对应的职业技能。
2. 队伍允许重复职业；同种 Buff 不叠加，只保留一个生效。
3. 一至四属性职业是常驻被动。
4. 五属性职业是主动技能，每场战斗 1 次。
5. 五属性主动必须 AP 等于最大 AP 才能使用，使用时消耗最大 AP。
6. 六属性职业是代价型主动，每回合 1 次；代价本场战斗持续，不重复抽取。
7. 当前战斗操作是 AP 驱动：单位只要仍有 AP，就可以被选中继续行动。因此“立即行动”不需要插入先攻队列，只需要恢复目标 AP。
8. 法术没有命中概念；非 INT 职业没有法术，仅远程攻击。
9. 旧版 `career_skill_configs.json` 和 `scripts/career_skills/*.lua` 属于 v0.8 职业大招体系，不能继续作为 v1 职业技能真相源。

## 当前代码状态

| 模块 | 当前状态 | 问题 |
|---|---|---|
| `ClassTitleResolver.cs` | 内置 63 个职业称号表 | 多个职业名仍是旧称号 |
| `CareerSkillRegistry.cs` | 从 `career_skill_configs.json` 读取旧主动大招 | 与 v1 分阶设计完全不一致 |
| `CareerSkillData.cs` | 只有主动/被动、AP、次数等基础字段 | 缺少“满 AP 才能用”“消耗最大 AP”“是否显示主动 UI”等规则字段 |
| `CareerSkillExecutor.cs` | 通过 LuaSkillBridge 执行旧 Lua 脚本 | v1 主动技能不应依赖旧 Lua |
| `CareerSkillResolver.cs` | 大量查询旧职业大招 Buff | 需要改为查询当前称号和 v1 运行时状态 |
| `CombatUI.cs` | 只显示 active 职业技能 | 方向正确，但需要按 v1 主动规则显示五/六属性 |
| `AIController` / AI 策略 | 会尝试释放 active 职业大招 | 需要只让 AI 使用 v1 五/六属性主动 |
| `UnitRuntimeState.cs` | 有通用运行时字段和旧职业标记 | 需要增加 v1 职业运行时状态 |
| 单元测试 | Godot 静态测试 runner，无 xUnit/NUnit | 需要新增职业技能专项测试套件 |

## 实现顺序

必须按以下顺序推进，避免 UI 拿到旧数据或执行器分发不到新技能。

### 1. 文档和数据一致性

目标：让代码里的职业称号和文档一致。

任务：
- 更新 `BladeHexCore/src/SkillTree/ClassTitleResolver.cs` 的 63 个称号表。
- 对照 `docs/职业名.md` 检查全部中文名。
- 保留英文名只在 `CareerSkillRegistry` 中维护，`ClassTitleResolver` 只负责中文称号。

验收：
- `ClassTitleResolver.GetTitleByFlags(flags)` 对 63 个非零 flags 都返回新中文名。
- `STR + DEX + CON + INT` 返回 `鏖战骑士`。
- `DEX + CON + INT + WIS + CHA` 返回 `万灵之约印`。

### 2. 职业技能数据模型

目标：让一个 `CareerSkillData` 能表达 v1 分阶规则。

文件：
- `BladeHexCore/src/SkillTree/CareerSkillData.cs`

新增字段：
- `RequiresFullAp`：AP 必须满才能使用。
- `ConsumesMaxAp`：使用时消耗最大 AP。
- `ShowInCombatUi`：是否显示在战斗职业主动按钮。
- 可选：`Tier` 或 `AttributeCount`，便于 UI 和测试判断分阶。

验收：
- 一至四属性技能：`Type=Passive`，`ShowInCombatUi=false`。
- 五属性技能：`Type=Active`，`LimitType=OncePerBattle`，`RequiresFullAp=true`，`ConsumesMaxAp=true`。
- 六属性技能：`Type=Active`，`LimitType=OncePerTurn`，`RequiresFullAp=false`，`ConsumesMaxAp=false`。

### 5. 主动职业技能执行

目标：五属性和六属性主动技能在游戏内可以正常使用。

文件：
- `BladeHexFrontend/src/View/Combat/CareerSkillExecutor.cs`
- `BladeHexFrontend/src/View/Combat/CombatManager.cs`
- `BladeHexFrontend/src/View/Combat/SkillTargetingInfo.cs`

通用规则：
- 被动职业不能执行主动。
- 五属性主动：
  - 当前 AP 必须等于最大 AP。
  - 成功后消耗最大 AP。
  - 每场战斗 1 次。
- 六属性主动：
  - 每回合 1 次。
  - 抽取一个未抽中过的代价。
  - 恢复自身 HP、法力、AP。
  - 三个代价均抽完后，本场战斗不能再使用。

主动技能逐项实现：

| 职业 | 目标 | 实现点 |
|---|---|---|
| 万灵之约印 | 目标友军 | 回满 HP、法力、AP；不插队；目标因 AP 满可继续被选中行动 |
| 山岳之王座 | 自身 | 挂 3 回合状态：不可移动、伤害 -60%、近战伤害 +60%、相邻敌人移动每格 +1 AP |
| 星界之裂隙 | 目标地图格 | 校验直线；移动到目标格；对路径敌人各执行一次攻击 |
| 荒芜之化身 | 自身 | 本场战斗状态：紧邻敌人护甲 100% 被穿透 |
| 铁血之律令 | 全体友军 | 所有友军获得 1 次“未命中转暴击”状态 |
| 孤星之刃影 | 紧邻敌人 | 校验周围 6 格仅有 1 个敌人；记录锁定目标；对该目标必中必暴，目标不能离开控制区 |
| 万象 | 自身 | 抽取无法移动/无法施法/无法攻击之一；恢复 HP/法力/AP；不重复抽取 |

验收：
- `5x 万灵之约印 + 1x 鏖战骑士` 能规则成立：五个万灵可以分别对同一个鏖战骑士使用主动。
- 万灵不会创建新的行动队列条目。
- 五属性 AP 不满时按钮可显示但执行失败，并返回明确原因。

### 6. 被动职业技能钩子

目标：把一至四属性被动接到现有战斗入口。

优先级 A：直接影响当前强 build 和核心战斗循环，必须第一批实现。

| 类型 | 职业 | 接入口 |
|---|---|---|
| 近战输出 | 战士、大宗师、战争领主、铁铸领主、战争之风、鏖战骑士 | `CombatResolver.ResolveAttack` |
| 移动触发 | 重战士、征服者、游骑兵、风语者、荒原之心、战誓者 | `MoveCommand`、`CombatMovementController` |
| 法术输出 | 法师、大贤者、秘院贤师、毁灭王冠、焰风之怒、天选者、唤星者、幻术师 | `SpellManager.CastSpell` 和各 ResolveSpell |
| 防御光环 | 守卫、守御者、铁幕领主、誓盾卫、磐石守护 | `Unit.GetEffectiveAc`、伤害结算前后 |
| 五属性联动 | 荒芜之化身、铁血之律令、孤星之刃影 | 攻击命中/暴击/穿甲流程 |

优先级 B：需要额外判定或边界，第二批实现。

| 类型 | 职业 | 接入口 |
|---|---|---|
| 远程掩体 | 游侠、鹰眼卫、影缄者 | `LineOfSight.GetPathPenalty` 调用前后 |
| 反击/额外攻击 | 决斗者、钢弦骑士 | `CombatResolver.ResolveAttack` 后处理 |
| 完整二次攻击 | 剑舞者 | `CombatManager` 级 OnMeleeHit 事件处理 |
| 资源互换 | 魔剑士、战法师、血契之环、敌法师 | 伤害、施法、回合结束 |
| 反法压制 | 静默之刃、敌法师 | `SpellManager.CanCastSpell`、法术伤害应用 |
| 单体锁定 | 孤刃之誓、孤星之刃影 | 攻击前条件、移动限制 |

#### 剑舞者专项实现

剑舞者的“额外攻击”不能作为普通静态数值钩子处理，必须走 `CombatManager` 级功能。

原因：
- 额外攻击需要找到“另一个相邻敌方目标”。
- 额外攻击必须执行完整 `CombatResolver.ResolveAttack` 管线，包括命中判定、暴击、伤害、护甲穿透、死亡、VFX、日志、士气/事件等后处理。
- 该额外攻击不能再次触发旋斩，否则会递归连锁。

实现拆解：
1. 在 `CombatResolver.ResolveAttack` 的近战命中结算成功后，发出一个可被 `CombatManager` 处理的 melee-hit 结果标记，而不是直接在静态方法内部寻找新目标。
2. `CombatManager` 在攻击结果处理阶段检查攻击者是否为剑舞者，并确认本次攻击不是旋斩额外攻击。
3. `CombatManager` 查询攻击者相邻敌方单位，排除本次原目标。
4. 按确定性规则选择目标，例如优先最低 HP，其次最近实例 ID，避免测试不稳定。
5. 调用 `CombatResolver.ResolveAttack(attacker, extraTarget, ..., isCareerExtraAttack: true)` 或等价上下文字段。
6. 在额外攻击上下文中禁止再次触发剑舞者旋斩。
7. 额外攻击结果走和普通攻击一致的反馈路径：伤害数字、死亡处理、日志、事件。

验收：
- 近战命中 A 后，若相邻存在另一个敌人 B，会对 B 进行一次完整攻击。
- 对 B 的攻击需要重新命中判定，不是自动伤害。
- B 的攻击不会再次触发旋斩。
- 原目标 A 不会被重复选为额外攻击目标。
- 没有另一个相邻敌人时不触发额外攻击。

优先级 C：可先暴露查询 API，再逐步接 UI/AI。

| 职业 | 原因 |
|---|---|
| 支配者 | 当前命中体系仍是 d20 + 百分比预览混合，需要确定最终命中修正口径 |
| 影缄者 | 需要半掩体/全掩体地形判定稳定 |
| 灵风秘庭 | 移动到法术伤害的跨动作状态需要明确本回合清理点 |

验收：
- 移动触发类能通过移动格数记录触发。
- 同种 Buff 不叠加。
- 自身型重复职业各自独立生效。
- 法术无命中逻辑，不给法术添加命中修正。

## 实施风险

1. 旧 `CareerSkillResolver` 内有大量 v0.8 Buff 查询，直接删除风险高；建议先新增 v1 方法，再逐步替换调用点。
2. 法术系统目前仍有 `AttackRoll` 分支，但设计上“法术没有命中概念”；职业被动不应新增法术命中加成。
3. “同种 Buff 不叠加”要依赖 Buff source/id 规则，不能通过重复应用同 id 同 source 叠层。
4. 万灵“立即行动”按当前 AP 驱动模型实现为恢复 AP，不新增先攻队列；如果以后改成严格先攻制，需要重新设计。
5. 部分被动依赖地图语义，例如半掩体、全掩体、连续同高度移动，需要确认 `HexCell` 已有足够字段，否则先做最小实现。
6. 剑舞者额外攻击是 CombatManager 级完整二次攻击，不是普通 OnMeleeHit 静态钩子；若直接在 `CombatResolver` 内递归调用，容易造成反馈丢失或无限递归。
7. `CareerSkillExecutor` 测试需要 Godot 运行时构造 `Unit` 节点，不能作为纯 Core 单元测试。
